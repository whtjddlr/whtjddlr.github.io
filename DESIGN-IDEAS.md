## 0. 소유자 취향 프로필 (아이디어 필터)

지금까지의 결정 이력. 새 아이디어가 이 프로필과 충돌하면 제안 전에 걸러낼 것.

| 확인된 취향 | 근거 |
|---|---|
| 감성적인 것 선호 — 종이 질감, 빛 번짐, 그레인, 따뜻한 무드 | "표지가 감성적이면 좋겠는데" 요청으로 커버 리디자인 |
| 움직임 선호 — 스택 쌓이는 느낌, 스크롤 상호작용 | "더 움직임이 있으면 좋겠다" 요청으로 모션 시스템 구축 |
| **하드한 도형 커서 거절 × 2** (그린 링, 반전 도트) | 현재는 수채 잉크 오브로 정착 |
| **워드마크 "just do it" 확정** (2026.07.08) | 나이키 차용 리스크 안내에도 유지 결정 — 바꾸지 말 것 |
| 네이비 테마 (2026.07 그린에서 전환) | 소유자 세션에서 직접 전환 |
| 아치형 사진 프레임 폐기 → 매트+크롭마크 | 크리틱의 "템플릿 트로프" 지적 수용 |

## 1. 테마·무드 변형 (큰 방향)

### 1-1. 다크 모드 — "밤의 서재" [중 · 중]
`prefers-color-scheme: dark` 대응. 단순 반전이 아니라 **네이비 종이(#10161F권)에
미색 잉크**로, 촛불 같은 웜 글로우 액센트. 그레인은 유지하되 밝기 반전.
- 주의: 증명사진·스크린샷은 밝은 이미지라 다크 배경 위에서 떠 보임 —
  매트 프레임 색을 다크용으로 별도 토큰화해야 자연스러움.
- 토큰이 이미 CSS 변수라 구현 비용은 색상 팔레트 설계가 대부분.

### 1-2. "기획서 문서" 컨셉 심화 [중 · 상]
현재의 문서형 레이아웃을 진짜 기획 문서의 관습으로 밀어붙이기:
- 섹션 번호(1. 2. 3.)와 문서 헤더("문서번호 CHO-2026-01 · 개정 2026.07")
- 각 프로젝트에 "승인: 배포됨 ✓" 결재란 위트
- 푸터를 문서 각주 스타일로
- PM 포트폴리오라는 정체성과 정확히 일치하는 유머 — 차별화 포인트.

### 1-3. 선박일지(Shipping Log) 컨셉 [상 · 중]
"배포 기록"의 은유를 문자 그대로: 날짜 레일을 항해일지 스타일로,
LIVE 배지를 등대 신호로. 파비콘 🚢과 이어짐.
- 주의: 은유가 과하면 유치해짐. 마이크로카피 수준에서만.

### 1-4. 계절/시간대 반응 [하 · 하]
방문 시각에 따라 배경 빛 번짐의 색온도가 미세하게 변함 (아침 쿨톤 → 저녁 웜톤).
JS 몇 줄. 알아채는 사람만 알아채는 이스터에그 수준.

## 2. 히어로 아이디어

### 2-1. 인용구 타이핑 연출 [하 · 중]
"잘하는 사람은 많으니…"가 타자기처럼 한 글자씩 써지고, 마지막에 "끝까지"에
형광펜이 칠해짐(현재 mark-swipe 유지). 커서 깜빡임까지.
- reduce-motion에서는 즉시 표시. 타이핑 속도는 총 1.5s 이내로.

### 2-2. "SHIPPED" 도장 연출 [중 · 중]
로드 시 히어로 어딘가에 도장이 쾅 찍히는 애니메이션 (scale 1.4→1 + 살짝 회전,
잉크 번짐 질감). 기획서 컨셉(1-2)과 결합하면 강력.
- 소유자가 하드한 도형을 싫어한 건 **커서** 맥락 — 도장은 별개지만 시안 먼저 보여줄 것.

### 2-3. 사진 마우스 시차(tilt) [하 · 하]
마우스 위치에 따라 히어로 사진이 1~2도 기울어짐. 현재 호버 리프트의 연장.
- perspective + rotateX/Y, lerp는 기존 커서 루프에 편승 가능.

### 2-4. 지표 카운트업 [하 · 하]
히어로 하단 지표(1위·5개)도 로드 시 카운트업. 수상 점수 카운트업과 같은 문법.
- "SSAFY 15기"는 숫자가 아니라 제외. 이미 리빌 스태거가 있어 과할 수 있음 — 하나만 고를 것.

## 3. 프로젝트 섹션 아이디어

### 3-1. 스티키 스택 카드 [상 · 상]
스크롤하면 프로젝트 카드가 화면 상단에 붙고 다음 카드가 그 위로 쌓이는 패턴
(`position: sticky` + 카드별 top 오프셋). 소유자가 원한 "스택 쌓이는 느낌"의 끝판.
- 주의: 카드 높이가 제각각이라 모바일에서 깨지기 쉬움. 데스크톱 한정으로 시작.
- 문서형(헤어라인) 레이아웃과 충돌 — 카드형으로 전환하는 큰 결정이 선행돼야 함.

### 3-2. 프로젝트별 포인트 컬러 [중 · 중]
각 서비스의 브랜드 색을 role-note 배경·LIVE 도트에만 소량 주입
(PlanMerge 보라, CARCH 청록, KOK 오렌지…).
- 주의: "액센트는 네이비 하나" 원칙과 충돌 — 원칙을 "전역 액센트는 네이비,
  프로젝트 카드 내부에서만 브랜드 색 허용"으로 개정할지 소유자 결정 필요.

### 3-3. 결정 로그 아코디언 [중 · 상]
각 프로젝트에 "이 결정, 왜 했나" 접힌 패널 — 버린 대안과 트레이드오프를
2~3줄로. 크리틱의 케이스 스터디 요구를 가볍게 흡수하는 디자인 장치.
- `<details>` 요소로 no-JS 안전. **콘텐츠는 소유자 입력 필요** (지어내기 금지).

### 3-4. Before/After 슬라이더 [상 · 중]
"기획 화면(피그마/스케치) ↔ 실제 배포 화면"을 드래그 슬라이더로 비교.
"기획을 그대로 만들었다"는 증거를 시각화 — PM 포지셔닝에 정확히 부합.
- 기획 단계 산출물 이미지가 있어야 함 (소유자 제공 필요).

### 3-5. 아이폰 프레임 목업 통일 [중 · 중]
세로 스크린샷을 CSS로 그린 폰 베젤 안에 넣어 완전 통일.
- 주의: CARCH는 이미 목업이 구워진 이미지라 이중 프레임 됨 — CARCH 원본을
  생캡처로 다시 받거나 CARCH만 제외하는 판단 필요. (기존 백로그 항목)

## 4. 인터랙션·모션 아이디어

### 4-1. 커서 잉크 트레일 [중 · 하]
현재 수채 오브가 지나간 자리에 옅은 잉크 자국이 1초간 남았다 마름.
- canvas 없이 DOM 파티클 5~6개 재활용으로 가능. 성능 주의 (rAF 루프 편승).

### 4-2. 커서 컨텍스트 라벨 [중 · 중]
프로젝트 링크 위에서 오브 옆에 "열기 ↗", 스크린샷 위에서 "확대"가 붙음.
클릭 가능성을 커서가 설명 — 안내적이고 PM스러움.
- 하드 도형 거절 이력과는 다름(텍스트 라벨) — 시안 먼저.

### 4-3. 스크롤 타임라인 진행선 [중 · 중]
프로젝트 섹션 왼쪽 날짜 레일을 세로선으로 잇고, 스크롤 위치에 따라
선이 네이비로 채워짐. "기록을 따라 내려가는" 감각 + 날짜 레일의 무정보 문제 개선.

### 4-4. 리더보드 스파크라인 [하 · 하]
수상 카드의 Private/Public 점수 옆에 미니 막대 2개 (0.976 vs 0.964 시각 비교).
inline SVG 몇 줄.

### 4-5. 픽셀 펫 이스터에그 [중 · 상(기억에 남음)]
갓생 퀘스트 펫의 픽셀 펫이 푸터에 서 있다가, 방문자가 페이지 끝까지
스크롤하면 걸어 나와 인사. "끝까지 읽은 사람에게 끝까지 만든 사람의 펫이 인사"
— 서사와 정확히 맞아떨어지는 개성. 채용 담당자가 기억할 디테일.
- 픽셀 스프라이트는 godlife-quest-pet 저장소 에셋 재활용 (라이선스 = 본인 것).
- reduce-motion 대응, 8×8 스프라이트 2~3프레임이면 충분.

## 5. 콘텐츠 표현 아이디어

### 5-1. 케이스 스터디 상세 페이지 템플릿 [상 · 상]
PlanMerge용 `planmerge.html`: 문제 가설 → 검토한 대안(버린 이유) →
트레이드오프 → 검증 → **다시 한다면**. 세로 타임라인 + 인용 블록 디자인.
- 크리틱 최우선 과제의 그릇. 콘텐츠는 소유자 입력 필요.
- "단일 페이지" 원칙의 예외가 됨 — AGENTS.md 개정 필요 (index는 단일 유지,
  상세 페이지는 별도 정적 HTML 허용 정도로).

### 5-2. 인터뷰 형식 자기소개 [중 · 중]
역량 섹션 아래에 Q&A 3문항 ("왜 PM인가요?" / "개발은 어디까지?" /
"어떤 팀에서 일하고 싶나요?"). 면접 예습을 먼저 보여주는 효과.
- 답변은 소유자가 써야 함. 디자인은 인용 대화 스타일.

### 5-3. 숫자 대시보드 스트립 [하 · 중]
"vitest 59 · 검수 요건 4종 · 스펙 문서 1,120행 · 커밋 N" 같은 숫자를
모노스페이스 티커로 한 줄. 사실 기반(README 대조 필수), 밀도 있는 인상.

### 5-4. 404 페이지 [하 · 하]
GitHub Pages는 `404.html` 지원. "이 페이지는 기획 단계에서 드롭됐습니다 —
백로그로 돌아가기 → [홈]" 같은 PM 유머 한 줄.

## 6. 우선 추천 조합 (제안)

가성비와 소유자 취향(감성+모션) 기준으로 고른다면:

1. **4-5 픽셀 펫 이스터에그** — 기억에 남는 개성, 본인 프로젝트 에셋 재활용이라 서사도 완벽
2. **2-1 인용구 타이핑** — 커버의 감성을 한 단계 올리는 저비용 연출
3. **4-3 스크롤 타임라인** — 모션 추가 + 날짜 레일 무정보 문제를 동시에 해결
4. **1-1 다크 모드** — 야간 열람 채용 담당자 경험 + 완성도 신호
5. **5-4 404 페이지** — 10분짜리 위트

구현 시: 시안(스크린샷)을 먼저 소유자에게 보여주고 진행할 것.
모든 모션은 AGENTS.md 5절의 모션 규칙(reduce-motion, no-JS 안전, 인쇄)을 따른다.
